# ilist v0.1.7

## Security release

This release closes the active-content and share-root authorization issues identified for the public beta.

- Same-origin HTML, SVG, XML, PDF, and unknown file bodies are forced to `application/octet-stream` attachments. Safe inline previews use an exact MIME allowlist and every same-origin file response receives a sandbox CSP, `nosniff`, `no-referrer`, and same-origin resource policy.
- Provider response headers are filtered. Only range and validator metadata is retained; provider `Set-Cookie`, content disposition, CSP, cache directives, and arbitrary diagnostic headers never reach clients.
- Share handles are encrypted, bound to their share root, and checked against live ancestry on every read. A moved item is denied even when an old handle is retained.
- Administrator login and share-password failures reserve a serialized verification slot before PBKDF2. Limits use HMAC-derived identities and do not store raw IP addresses or passwords.
- New mounts are private by default at the API, domain, UI, and D1 schema layers. Publishing requires explicit confirmation.
- New password hashes use `pbkdf2-sha256:600000`; supported legacy hashes are verified during the transition and share hashes upgrade after a successful verification.

## Deployment order

1. Export the remote D1 database outside the repository:

   ```bash
   npx wrangler d1 export ilist-db --remote --output /tmp/ilist-db-before-v0.1.7.sql
   ```

2. Generate and rotate `ADMIN_PASSWORD_HASH`. The command never accepts a password argument; it reads a hidden TTY prompt or exactly one stdin line and writes only the hash to stdout.

   ```bash
   npm run hash-password
   npx wrangler secret put ADMIN_PASSWORD_HASH
   ```

3. Apply all remote D1 migrations before deploying the Worker:

   ```bash
   npx wrangler d1 migrations apply ilist-db --remote
   ```

   This release adds `0015_auth_rate_limits.sql` and `0016_mounts_private_default.sql`. `0015` adds hashed authentication-rate-limit records. `0016` rebuilds `mounts` with `is_public DEFAULT 0` while explicitly preserving every existing mount's publication state and all dependent credentials, OAuth state/leases, upload sessions, and shares. The native `R2` compatibility mount remains explicitly public.

4. Set the canonical origin and deploy Worker plus Assets together:

   ```bash
   npx wrangler secret put PUBLIC_ORIGIN
   # Enter: https://ilist.chius.cc
   npm run deploy
   ```

5. Smoke-test the deployed application: HTML and SVG downloads are attachments, the sixth failed administrator login is rate-limited, a moved shared child is denied through its old handle, new mount creation is private, and an explicitly published mount is visible to a guest.

## Custom-domain and OAuth cutover

`wrangler.jsonc` declares the official custom-domain route `ilist.chius.cc` and deliberately leaves the default `workers.dev` hostname enabled during cutover. Before deployment, ensure `ilist.chius.cc` belongs to the authenticated Cloudflare account and proxy is enabled.

1. Add these redirect URIs to the existing provider applications:

   ```text
   https://ilist.chius.cc/api/admin/oauth/onedrive/callback
   https://ilist.chius.cc/api/admin/oauth/google/callback
   ```

2. Retain both existing `https://ilist.chius.workers.dev/.../callback` redirect URIs until signing in, authorizing, reconnecting, and provider callbacks have been verified at `ilist.chius.cc`.
3. Set `PUBLIC_ORIGIN` exactly to `https://ilist.chius.cc`; it must have no trailing slash.
4. Only after the custom domain has passed the smoke tests should the legacy `workers.dev` OAuth callbacks be considered for removal.

## Rollback

Roll back Worker code and Assets to the previous known-good Worker version without rolling back D1 migrations. `0015` is additive. `0016` preserves existing data and introduces a private default, so restoring a D1 export is reserved for data corruption or a failed migration, not ordinary Worker rollback. If a restoration is required, stop writes, restore the export, verify foreign keys and mount credentials, then redeploy a compatible Worker.
