# ilist v0.1.7

## Security release

This release closes the active-content and share-root authorization issues identified for the public beta.

- Same-origin HTML, SVG, XML, PDF, and unknown file bodies are forced to `application/octet-stream` attachments. Safe inline previews use an exact MIME allowlist and every same-origin file response receives a sandbox CSP, `nosniff`, `no-referrer`, and same-origin resource policy.
- Provider response headers are filtered. Only range and validator metadata is retained; provider `Set-Cookie`, content disposition, CSP, cache directives, and arbitrary diagnostic headers never reach clients.
- Share handles are encrypted, bound to their share root, and checked against live ancestry on every read. A moved item is denied even when an old handle is retained.
- Administrator login failures reserve two serialized verification slots before PBKDF2: five failures per minute for an IP plus the existing five failures per minute for that IP and normalized username. Share-password failures remain independently limited to ten per minute per IP and share. Limits use HMAC-derived identities and do not store raw IP addresses or passwords.
- New mounts are private by default at the API, domain, UI, and D1 schema layers. Publishing requires explicit confirmation.
- New password hashes use `pbkdf2-sha256:600000`; supported legacy hashes are verified during the transition and share hashes upgrade after a successful verification.

## Deployment order

1. Export the remote D1 database outside the repository:

   ```bash
   npx wrangler d1 export ilist-db --remote --output /tmp/ilist-db-before-v0.1.7.sql
   ```

2. Do **not** rotate `ADMIN_PASSWORD_HASH` before the first v0.1.7 deployment. v0.1.7 accepts the existing legacy `pbkdf2` secret. Retain a protected, out-of-repository break-glass copy of that legacy hash before deployment; do not put it in a shell history or release notes.

3. Apply all remote D1 migrations before deploying the Worker:

   ```bash
   npx wrangler d1 migrations apply ilist-db --remote
   ```

   This release adds `0015_auth_rate_limits.sql` and `0016_mounts_private_default.sql`. `0015` adds hashed authentication-rate-limit records. `0016` rebuilds `mounts` with `is_public DEFAULT 0` while explicitly preserving every existing mount's publication state and all dependent credentials, OAuth state/leases, upload sessions, and shares. The native `R2` compatibility mount remains explicitly public.

4. Complete the custom-domain preflight below, set the canonical origin, and deploy Worker plus Assets together:

   ```bash
   npx wrangler secret put PUBLIC_ORIGIN
   # Enter: https://ilist.chius.cc
   npm run deploy
   ```

5. Smoke-test the deployed application: HTML and SVG downloads are attachments, the sixth failed administrator login is rate-limited (including after changing usernames), a moved shared child is denied through its old handle, new mount creation is private, and an explicitly published mount is visible to a guest.

6. Only after the first v0.1.7 smoke test passes, rotate the administrator secret to the current hash format and repeat the administrator login smoke test:

   ```bash
   npm run hash-password
   npx wrangler secret put ADMIN_PASSWORD_HASH
   ```

## Custom-domain and OAuth cutover

`wrangler.jsonc` declares the official custom-domain route `ilist.chius.cc` and deliberately leaves the default `workers.dev` hostname enabled during cutover. A Workers Custom Domain is not a manually proxied DNS record: Cloudflare creates and manages the DNS route and TLS certificate after deployment.

1. Before deployment, confirm that `ilist.chius.cc` belongs to the authenticated Cloudflare zone and has no conflicting CNAME, A, AAAA, or other DNS record. Remove or migrate a conflicting record before creating the Custom Domain.
2. After deployment, wait for the Custom Domain certificate to become active, then verify `https://ilist.chius.cc` completes TLS validation and serves the Worker. Do not continue OAuth cutover while the certificate is pending or HTTPS is unavailable.

3. Add these redirect URIs to the existing provider applications:

   ```text
   https://ilist.chius.cc/api/admin/oauth/onedrive/callback
   https://ilist.chius.cc/api/admin/oauth/google/callback
   ```

4. Retain both existing `https://ilist.chius.workers.dev/.../callback` redirect URIs until signing in, authorizing, reconnecting, and provider callbacks have been verified at `ilist.chius.cc`.
5. Before changing `PUBLIC_ORIGIN`, finish or cancel in-progress `workers.dev` OAuth authorizations and wait up to ten minutes for their states to expire. The new origin intentionally rejects callbacks bound to the old origin.
6. Set `PUBLIC_ORIGIN` exactly to `https://ilist.chius.cc`; it must have no trailing slash.
7. Only after the custom domain has passed the smoke tests should the legacy `workers.dev` OAuth callbacks be considered for removal.

## Emergency abort and rollback

Do **not** use v0.1.6 as a normal public rollback target: it reintroduces the stored-content and share-root vulnerabilities fixed here and cannot authenticate with a rotated `pbkdf2-sha256` administrator hash. Keep `0015` and `0016` applied; they are not rolled back for Worker-code recovery.

If a critical issue occurs, first stop public traffic with Cloudflare Access, a restrictive WAF/route policy, or maintenance access controls. Prefer a fix-forward deployment on v0.1.7. A pre-rotation emergency rollback to old code is only acceptable while public traffic remains restricted and only after restoring the retained legacy `ADMIN_PASSWORD_HASH`; it is a controlled diagnostic measure, not a public-service state. After the new hash format has been deployed, do not roll back to v0.1.6. Restore the D1 export only for data corruption or a failed migration, then verify foreign keys, mount visibility, credentials, and share records before redeploying a compatible v0.1.7 Worker.

## Residual provider risk

OneDrive and Google Drive share checks verify live ancestry before provider metadata or download requests. A concurrent provider-side move can therefore create a narrow TOCTOU window for one request. This cannot be made atomic through those provider APIs; restrict provider write access for sensitive public shares and disable the share when immediate revocation is required.
